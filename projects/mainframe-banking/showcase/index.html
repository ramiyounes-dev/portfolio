HOW IT WORKS

THE IDEA

This demo simulates a mainframe banking batch processing system built with GNU COBOL. In real banks, transactions are not applied one by one — they are collected into batches, sorted, then posted in bulk using an efficient algorithm called appareillage (a lockstep walk of two sorted files).

The system is composed of 7 COBOL programs compiled to native binaries, a Flask web layer that calls them via subprocess, and a terminal-style UI. This showcase page runs entirely in the browser with no backend — it simulates the same logic client-side.

BATCH LIFECYCLE

Every transaction passes through three phases, each color-coded in the ledger:
1
PENDING — Raw transaction appended to an unsorted buffer file (pending.dat). No account balances are touched yet.
2
SORTED — The COBOL SORT verb reorders the batch by account number, then by timestamp. This is what makes the merge efficient.
3
COMMITTED — The sorted batch is walked in lockstep with the indexed accounts file. Each account balance is updated, and the transaction is written to the permanent ledger.

THE ALGORITHM: APPAREILLAGE

UPDATE-BALANCES reads the sorted transaction file sequentially. For each transaction, it does a keyed read on the indexed accounts file, adds the amount to the balance, rewrites the account record, and appends a committed record to the transaction log — all in a single pass at O(N+M) complexity. This is the same pattern mainframes use to process millions of records nightly.

HOW TO USE THIS DEMO

1
Add transactions — Click + 1 Random Transaction or + 10. New rows appear in red at the bottom of the ledger.
2
Sort the batch — Click Sort Batch. Red rows turn orange and are reordered by account number. The system log shows the COBOL SORT statement.
3
Post to accounts — Click Update Accounts & DB. Orange rows turn green, balances update, and the batch divider appears. Changed balances are highlighted in the results panel.
4
Inspect a batch — Click any BATCH #N divider in the ledger to see per-account deltas and the full transaction list for that batch.
5
Manage accounts — Click a row in the accounts ledger to select it, then use + New Account, Delete Account, or Check Transactions for full history.

ERROR HANDLING

If a transaction references an account that does not exist or is closed, the system rejects it. ADD-TRANSACTION returns an error. UPDATE-BALANCES logs a warning and flags the transaction as ALERT (yellow) in the ledger — the transaction is skipped, not committed. This makes orphaned or invalid transactions immediately visible for investigation.

FILE ORGANIZATION

Five data files, each mirroring a mainframe VSAM dataset type:
1
accounts.dat — INDEXED (KSDS), 120-byte records keyed on ACCT-NUMBER. Supports random reads by key and sequential scans.
2
transactions.dat — SEQUENTIAL (ESDS), 150-byte records. Append-only permanent ledger of committed transactions.
3
pending.dat — SEQUENTIAL (ESDS), 150-byte records. Unsorted buffer for raw transactions awaiting the SORT phase.
4
sorted-staging.dat — SEQUENTIAL (ESDS), 150-byte records. Output of COBOL SORT, input to the appareillage merge.
5
batch-state.dat — SEQUENTIAL, 80-byte single-record control file tracking batch lifecycle (IDLE → SORTED → POSTING → IDLE).
All monetary values use COMP-3 packed decimal (PIC S9(13)V99 COMP-3 = 8 bytes) — the same format used on IBM mainframes to avoid floating-point rounding.

ACCESS PATTERNS

Each COBOL program opens only the files it needs with the minimum access mode required:
▶
CREATE-ACCOUNT — DYNAMIC read on accounts (READ NEXT to find max key, then WRITE new record)
▶
ADD-TRANSACTION — RANDOM read on accounts (verify existence, reject if not found), EXTEND on pending
▶
SORT-BATCH — Reads pending sequentially, uses COBOL SORT verb with SD work file, writes sorted-staging
▶
UPDATE-BALANCES — Reads sorted-staging sequentially; for each transaction, does a random keyed read on accounts. If found and active: updates balance via REWRITE and appends to transactions. If missing or closed: logs warning and flags the transaction as ALERT
▶
QUERY-ACCOUNT — RANDOM read on accounts by key, sequential scan of transactions for history
▶
DELETE-ACCOUNT — RANDOM read + REWRITE on accounts (mark CLOSED), EXTEND on transactions (closure record)
